# 体裁模板 · analytical（分析 / 评审 / 报告 / 多步推理）

适用：竞品分析、技术选型对比、code review 评语、根因分析、行业研究——"给人读、但要论证严密"的产出。
结构哲学：分析质量来自**框架与证据规则**，不是人设。骨架重心在"分析框架"（维度即契约——它是本体裁的 schema 等价物）与证据链；输出是**结论先行**的人读轮廓，不是 JSON。

## 骨架

```markdown
# 任务与视角
以 <谁的立场 / 什么判断标准> 分析 {{对象}}，要回答的核心问题：{{question}}。

# 分析框架（本体裁的主体——维度即契约，锁维度 = 锁输出空间）
按以下维度逐一分析，不得增删维度：
1. <维度 A：看什么、用什么标准判好坏>
2. <维度 B：…>
3. <维度 C：…>
- 权衡规则：<维度间冲突时的优先级（如安全 > 成本 > 体验）>

# 证据规则（防幻觉的主力）
- 每个判断必须挂证据：<引用来源 / 数据 / 原文位置 / 行号>
- 每条陈述标注类别：【事实】（有出处）/【推断】（由事实导出，写出推理链）/【假设】（未验证，标注如何验证）
- 证据不足 → 写"证据不足，需补充 X 才能下判断"，不许补脑

# 输出结构（人读轮廓，非 schema）
1. TL;DR：结论一句话 + 最关键的 1-3 个依据（≤{{N}} 字）
2. 逐维度分析：每维度 = 判断一句话 + 证据 + 置信度（高/中/低）
3. 风险与反例：什么情况下上述结论会被推翻
4. 建议动作：按优先级排列，每条可执行

# 自检（输出前）
- 结论是否直接回答了核心问题（而非答了个相邻问题）？
- 有没有不挂证据的判断？反例是否认真找过（至少一条）？
```

## 填写要点

- **分析框架是第一公民**：维度从 `IR.intent.success_criteria` 与 domain 推导，3-6 个为宜；"不得增删维度"这句必须写——否则模型会自由发挥出一篇散文。
- **证据三分类**（事实/推断/假设）比"不要编造"有效一个数量级：它把防幻觉从禁令变成了输出格式的一部分。
- **结论先行**：TL;DR 放最前。读者是人，人先要答案再要论证——这与 extract 体裁"先想后出"的顺序刚好相反，别搞混。
- 置信度三档（高/中/低）写进每个维度，给读者校准信任用。
- 多步推理任务在 Workflow 维度上加"对照框架自检"步骤即可，不需要也不应该引入 schema。

## 反例自检

- 模型可以自由决定分析哪些方面 → 框架没锁住，未过。
- 通篇判断无一处引用/出处 → 证据规则没生效，未过。
- 输出是 JSON 或表格堆砌 → 人读体裁错配成了 machine 形态，回查 `output_contract.consumer`。
