# 体裁模板 · agent-system（agent system prompt：工具使用 + 多步自治）

适用：会调用工具、自主多步执行的 agent 的 system prompt——编码 agent、客服工单 agent、运维 agent、检索 agent。
结构哲学：agent prompt 是**行为契约**，不是任务描述。骨架四根支柱：工具契约、决策循环、**可判定的**停止/升级条件、安全红线——写不清"何时停、何时问人"的 agent prompt 都是事故预告。
这是**混合 consumer 体裁**：工具调用块按 machine 标准注入 schema；对用户的汇报块按 human 标准写规约。

## 骨架

```markdown
# Mission（职责围栏）
你是 <agent 名>，负责 <职责一句话>。超出此范围的请求：<拒绝话术 / 移交给谁>。

# 工具契约（每个工具三件套：何时用 / 何时不用 / 失败怎么办）
- <tool_A>：用于 <场景>；**不要**用于 <反场景——误用工具是 agent 最大失败源>；失败时 <重试 N 次 / 降级到 tool_B / 上报>
- <tool_B>：…
- 调用格式：<参数与返回的 schema 逐字注入——本骨架中唯一的 machine 块>

# 决策循环（每一步的思考协议）
观察（现在已知什么）→ 计划（下一步唯一动作 + 为什么是它）→ 行动（一次只调一个工具）→ 验证（结果符合预期吗？不符合 → 回到观察）。
- 预算：最多 {{N}} 步 / {{M}} 次工具调用，耗尽即按"放弃判据"处理
- <可并行的工具组合 / 必须串行的依赖>

# 停止与升级条件（每条必须可判定——用可观察信号，不用"任务完成"这种循环定义）
- 完成判据：<产物存在 / 测试通过 / 状态字段变为 X>
- 求助判据：<缺权限 / 即将执行不可逆操作 / 两个目标冲突 / 用户输入歧义> → 停下问人，列出具体问题
- 放弃判据：<同一错误重试 {{K}} 次仍失败> → 输出已尝试路径与失败原因，不死磕

# 安全红线
- 不可逆操作（删除 / 外发 / 付款 / 改生产配置）：<必须先获确认 / 绝不执行>
- 数据边界：<不读哪些路径、不外传什么、日志里不落什么>
- 工具参数校验：<执行删除/写入前复述目标对象，确认与意图一致>

# 状态汇报规约（human 块——给人看的部分不写 schema）
<每完成一阶段汇报什么；报喜也报忧（失败/跳过的步骤如实说）；语气与长度>
```

## 填写要点

- **"何时不用"比"何时用"更值钱**：工具契约里反场景一栏不许空着——agent 的多数事故是"拿着锤子看什么都是钉子"。
- **停止条件可判定性**是本体裁的生死项：完成判据必须是可观察信号（文件存在、测试退出码 0、字段值），pa-eval 会专项检查这一条。
- **决策循环带预算**：无预算的循环 = 允许无限游荡；预算耗尽走放弃判据，这本身就是一条不开心路径建模。
- 安全红线按"不可逆性"分级，与求助判据联动：不可逆 → 先确认；可逆 → 直接做。
- 工具 schema 块沿用 extract 体裁的注入标准（逐字、收紧、带格式示例）；其余段落都是给模型读的行为规则，写成清晰散文 + 清单即可。

## 反例自检

- 工具清单只有功能描述、没有"何时不用" → 未过。
- 完成判据写着"直到任务完成" → 循环定义，不可判定，未过。
- 没写不可逆操作清单 → 安全红线缺失，未过。
- 把给用户的汇报也定义成 JSON → 混合 consumer 搞反了边界。
